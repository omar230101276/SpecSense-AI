Technical Report: Intelligent OCR & AI System for Cable Specification Extraction (SpecSense AI)
Module: Validation Engine  
Project: SpecSense AI  

--------------------------------------------------------------------------------

1. Introduction

Overview of the SpecSense AI System
SpecSense AI is an advanced, modular system designed to extract, validate, and analyze electrical cable specifications. It automates the interpretation of cable datasheets and engineering drawings. The system integrates computer vision for text extraction, natural language processing for parameter identification, and a rigorous electrical engineering rule engine to ensure data integrity.

Role of the Validation Module
The Validation Module serves as the critical "gatekeeper" of the SpecSense AI system. Once the OCR Reader extracts raw technical specifications, the Validation Module analyzes this data to ensure it complies with real-world physical laws, electrical engineering standards, and basic common sense. It prevents OCR hallucinations or physically impossible specifications from being passed to the downstream keyword generation and technical assistant modules.

--------------------------------------------------------------------------------

2. System Architecture (Validation Module)

High-Level Architecture
The validation pipeline is built on a deterministic rule-based engine encapsulated in the `CableValidator` class. It sits squarely between the OCR/Extraction layer and the final output layer.

Data Flow:
1. Input Reception: Receives a structured JSON dictionary of extracted parameters (e.g., voltage, conductor size, insulation type) from the OCR post-processor (`SpecCorrector`).
2. Value Parsing & Normalization: Extracts raw numeric values from alphanumeric strings (e.g., parsing `450/750V` into min/max voltage floats).
3. Rule Execution: Passes the normalized data through a sequential series of domain-specific engineering checks (Rules 1 through 11).
4. Status Classification: Evaluates cumulative rule violations to categorize the specification status as `READY`, `NOT READY`, or `UNVERIFIABLE`.
5. Output Generation: Returns a comprehensive validation dictionary containing the status, missing data alerts, and a detailed list of engineering violations.

Integration with Other Modules
The validator ensures that downstream modules only receive physically possible data. If the validation is successful (`READY`), the pipeline automatically triggers the Keyword Generation Module and the Technical Assistant. If it fails, the process is terminated to prevent cascading errors in subsequent AI analysis.

--------------------------------------------------------------------------------

3. Validation Logic & Design

The module employs a strict rule-based approach driven by domain-specific electrical constraints.

Types of Rules Implemented:
* Range Validation: Enforces strict upper and lower bounds. For example, operating temperatures must fall within realistic ranges (-40 to 105 degrees Celsius).
* Format & Consistency Validation: Rejects fractional conductor counts (e.g., "4.5 cores" is physically impossible) and flags ambiguous cable types (e.g., hybrid Fiber-Optic/Power cables).
* Physics & Density Constraints: Calculates the current density (Amps per mm2). It rejects current ratings that yield > 30 A/mm2 (physically incompatible/melting risk) or < 0.1 A/mm2 (likely an OCR truncation error).
* Standard Compliance (IEC): Verifies that extracted conductor sizes match standard IEC cross-sections (e.g., 1.5, 2.5, 4.0 ... up to 1000.0 mm2) within a 5% tolerance margin.
* Material & Voltage Compatibility: Enforces high-voltage safety rules. For instance, PVC insulation is explicitly rejected for voltages exceeding 3.3kV (must be XLPE), and any cable over 1kV strictly requires a verified insulation type to be classified as safe.

--------------------------------------------------------------------------------

4. Technologies & Tools Used

- Programming Language: Python 3.x
- Rule Engine Design: Custom object-oriented logic built into the `CableValidator` class.
- Libraries Used: Built-in `re` module for advanced regex-based float extraction and unit parsing. `json` and `pandas` for handling I/O and reporting in the outer wrapper (`valid.py`).
- Data Structures: Heavily relies on Python dictionaries to maintain rulesets (valid insulation types, valid armor types, invalid materials) and schemas for the input/output payloads.

--------------------------------------------------------------------------------

5. Validation Workflow

1. Input Format: A structured dictionary of OCR outputs, which have already been passed through the `SpecCorrector`.
2. Step-by-Step Validation:
   - Missing/Unverifiable Data Check: Scans for flags left by the OCR corrector indicating unreadable fields.
   - Sequential Rule Processing: Runs through 11 distinct engineering checks, appending violation strings to an internal error array if triggered.
3. Error Detection and Classification:
   - "UNVERIFIABLE": Critical data (like Voltage) is missing, or the OCR engine explicitly marked fields as unreadable.
   - "NOT READY": Data was extracted but violates one or more engineering rules (e.g., "PVC Insulation cannot be used for High Voltage").
   - "READY": All data is present and complies with standard physics and IEC sizing.
4. Output Format: A JSON structure detailing the `valid` boolean flag, `status` string, `errors` list, and `missing` data list.

--------------------------------------------------------------------------------

6. Error Handling & Correction Strategy

The Validation Module operates strictly as an auditing engine; it flags errors rather than guessing corrections. The actual data correction (like "S" to "5" or "NxS" splitting) is deliberately handled upstream by the `SpecCorrector` before validation occurs.

Examples of Handled OCR Errors:
- Absurd Current Ratings: If OCR misses a digit (reading "2A" instead of "200A" for a 150mm2 cable), the validator catches this via the minimum density rule (Rule 3b) and flags it as an OCR Error rather than accepting a dangerously low capacity.
- Decimal Misplacements: If OCR reads a fractional core count (e.g., "3.5 cores"), Rule 5 instantly rejects it.
- Non-Electrical Materials: If OCR misinterprets background noise as "GLASS" or "WOOD", Rule 4 rejects it against the whitelist of valid materials.

--------------------------------------------------------------------------------

7. Improvements & Modifications

- Initial Approach vs. Improved Version: The initial validator merely checked if data types were strings or numbers. The improved version incorporates deep electrical engineering physics, actively calculating current densities and checking IEC standard tables.
- Robust Float Parsing: Early versions crashed when parsing complex voltages like `0.6/1kV`. The current regex-based float parser intelligently extracts all numerical voltages and applies unit multipliers (kV to V) before calculating ratios to prevent false positives.
- Modularity: The core logic was decoupled from the CLI execution (`valid.py`), moving the `CableValidator` into `src.validation.py` to allow seamless integration with the web backend and other tools.

--------------------------------------------------------------------------------

8. Evaluation & Performance

- Effectiveness: The rule engine effectively catches 100% of physically impossible combinations generated by OCR hallucinations.
- Reduction in OCR Errors: By flagging impossible physical ratios (like 500A on a 1.5mm2 wire), the validator prevents catastrophic downstream analysis, ensuring the AI assistant only gives advice on real-world parameters.
- Example Correction Workflow:
  * OCR reads: `{"voltage": "11kV", "insulation": "PVC"}`
  * Validation Output: `Status: NOT READY. Errors: ["10. Material: PVC Insulation cannot be used for High Voltage (11kV). Must be XLPE."]`
- Reliability: Because the rules are deterministic and based on hard electrical standards, the system's decisions are highly reliable and easily interpretable by human engineers.

--------------------------------------------------------------------------------

9. Advantages

- Accuracy Improvement: Eliminates mathematical hallucinations and enforces reality checks on optical data.
- High Reliability: Builds trust in the overall SpecSense AI system by ensuring no dangerous or non-compliant cable specifications are passed to the user as "valid."
- Extensibility: The dictionary-based rule definitions (e.g., `valid_armor`, `standard_sizes`) make it trivial to add new materials or update standard sizes without rewriting core logic.

--------------------------------------------------------------------------------

10. Limitations

- Rigid Rule Bounds: Highly specialized, custom-manufactured cables that fall outside standard IEC cross-sections (e.g., a custom 130mm2 cable) will be incorrectly flagged as invalid by Rule 9.
- Dependence on Predefined Rules: The system cannot validate what it does not know. If a new, revolutionary insulation material is introduced to the market, the system will reject it until the code is manually updated.
- Incomplete Data: If the OCR fails to capture both Current and Conductor Size, the vital current density check (Rule 3) cannot be executed, potentially allowing subtle errors through.

--------------------------------------------------------------------------------

11. Future Enhancements

- Adaptive Validation & ML: Integrating a machine learning anomaly detection model to learn from historical, manually verified datasheets, allowing the system to recognize valid but uncommon cable specs.
- Expanded Standard Coverage: Incorporating a toggle system to switch between IEC, NEC (AWG sizes), and BS standards depending on the geographical region of the user.
- User Feedback Loop: Implementing a UI feature where engineers can "Override and Approve" a flagged validation, which the system logs to continuously improve its rule thresholds.

--------------------------------------------------------------------------------

12. Conclusion

The Validation Module acts as the crucial safety and reality check within the SpecSense AI ecosystem. By combining deterministic Python logic with strict electrical engineering principles, it successfully filters out OCR artifacts and impossible specifications. This ensures that all downstream AI analysis, keyword generation, and technical advice are based solely on physically viable and standard-compliant cable data, establishing the system as a reliable tool for engineering professionals.
